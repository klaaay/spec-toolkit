# 按接口契约实施前端代码

这个技能承接旧的 `fe-definition-gen.run`，专门处理“接口定义已经存在，如何把它可靠地接入前端”的问题。

## 前置条件

- 已有权威接口定义，例如接口标识、接口文档链接、OpenAPI 描述、curl 示例或示例请求与响应
- `.specify/memory/fe-rule.md` 已存在且可信
- 当前任务已有已确认的实施文档，或用户给出的目标已经足够具体

如果前端宪法缺失，先使用 `frontend-rules-bootstrap`。
如果需求和页面行为仍不明确，先使用 `superpowers-writing-plans`。

## 工作流

1. 读取 `.specify/memory/fe-rule.md`。
   先确认网络层模式、类型组织方式、目录规范、测试策略和命名约束。

2. 解析用户输入中的契约信息。
   至少要明确接口标识、调用场景、目标页面或组件，以及需要重点关注的数据字段。

3. 获取权威接口定义。
   优先使用当前会话里可用的接口契约来源，例如 OpenAPI、接口文档、示例请求与响应或仓库内已有声明，不要依赖手写猜测。需要拿到请求参数、响应结构、错误码和鉴权要求。
   如果当前环境里找不到足够权威的契约来源，必须要求用户主动提供接口文档、OpenAPI、curl、示例响应或关键字段说明，再继续实施。

4. 对齐现有前端实现。
   如果接口声明、service、hook 或类型已经存在，先做比对，再决定是增量更新还是新建文件。

5. 先落地契约层。
   包括请求类型、响应类型、错误类型、接口声明、service 或 hook 封装。

6. 再落地页面集成层。
   把新的契约接入页面、组件或状态流，补齐加载态、空态、失败态和必要的用户反馈。

7. 若仓库已有测试模式，就补上最小必要验证。
   覆盖成功路径、失败路径和关键边界，而不是只改业务代码不校验。

8. 运行最相关的验证命令。
   至少覆盖类型检查、lint，以及目标应用相关测试或构建。

## 关键原则

- 以权威接口定义为准，不凭记忆写类型
- 先对齐现有封装，再决定是否新建抽象
- 契约层和页面层分开处理，避免所有逻辑揉进单一组件
- 如果接口文档和前端宪法冲突，先显式暴露冲突，不要静默偏离
