# Recommended Rules

本文档记录 `aged-*` 项目推荐遵守的稳定规范。这些规则覆盖模块组织、接口设计、数据访问、认证授权、安全、可靠性与交付检查，不要求模板外项目逐字照搬目录。

## 模块组织约定

1. 后端优先采用 `module-first`。目录按业务模块组织，不按 controller、service、repository 全局分层平铺。
2. 默认实现路径优先是 `router.py -> service.py -> Session + ORM model`。`service.py` 是默认业务入口。
3. `repository.py` 只在查询复杂或需要复用查询拼装时引入，不要为了分层好看预先创建空文件。
4. `platform` 只放运行时基础设施，`shared` 只放轻量公共能力，不能成为业务逻辑回收站。
5. 应用生命周期资源，如 Redis 连接，应在 `lifespan.py` 中初始化和清理。

## 接口设计与错误处理

1. 后端优先采用 Python Web 框架，如 Flask、FastAPI，并保持 HTTP 层与业务层分离。
2. 接口应遵循 RESTful 风格。资源路由语义清晰，HTTP 方法使用正确：`GET` 查询、`POST` 创建、`PUT/PATCH` 更新、`DELETE` 删除。
3. HTTP 状态码必须语义化：
   - `200` 成功
   - `201` 创建成功
   - `400` 参数错误
   - `401` 未认证
   - `403` 无权限
   - `404` 资源不存在
   - `500` 服务异常
4. router 返回值统一使用 `success_response(data)` 包裹，保持成功 `{"data": ...}` / 失败 `{"error": {"type": ..., "message": ...}}` 的对称格式。
5. 必须有全局异常处理。服务不能因预期外异常直接崩溃，也不能把堆栈或原始错误直接暴露给客户端。

## 数据库与 ORM

1. 表结构应明确主键、外键、字段类型、非空约束与唯一约束。
2. 高频查询字段和排序字段必须评估索引，例如 `user_id`、`category_id`、`created_at`、`price`。
3. 列表接口涉及关联数据时，必须显式避免 N+1 查询，优先使用 eager loading，如 `joinedload`、`selectinload`。
4. ORM 默认采用成熟方案，如 SQLAlchemy、Tortoise。
5. 查询必须走参数化或 ORM 表达式，不允许字符串拼接 SQL。
6. 多表写操作必须放在事务内，失败时显式回滚。

## 认证、授权与安全

1. 密码必须使用加盐哈希，如 `bcrypt`、`argon2`。禁止明文存储，禁止简单 `MD5`，禁止可逆加密代替密码哈希。
2. 登录态保持方式应显式定义，采用 `Session` 或 `JWT` 均可，但项目内应保持一致。
3. 权限控制不能只做角色判断。除 `RBAC` 外，修改、删除、敏感读取等操作还要校验资源归属或作用域。
4. SQL 注入防护是基线要求，所有动态条件都必须参数化。
5. 认证失败、权限不足、资源不存在时，状态码必须分别返回 `401`、`403`、`404`，不能混用。

## 性能与可靠性

1. 列表接口必须做分页，并验证边界。`page >= 1`，`limit` 必须有上限。
2. 不允许无界查询、无上限分页或一次性返回超大结果集。
3. 有明显热点读场景时，应评估 Redis 缓存，如分类数据、热点详情、聚合结果。
4. 关键操作应具备基本异常恢复能力，例如重试、降级、幂等保护或补偿策略。
5. 错误日志、关键链路日志与异常上下文应可追踪。

## 前端对接规范

1. 请求能力统一收口到 `service`，不要额外并行创建新的 `lib` 请求入口。
2. 默认采用统一的 `axios` client、拦截器和错误归一化。
3. 页面与 hooks 只消费模块 service，不直接拼请求。
4. service 模块无需手动拆包 `{"data": ...}` 信封，interceptor 已自动处理。
5. 静态跨端元信息尽量单一来源。

## 部署与交付检查

1. 起步项目时，优先补跑 `node ./scripts/verify-init-project.mjs`。
2. Docker 镜像应优先采用多阶段构建，避免把构建工具链和开发依赖带入运行时镜像。
3. Nginx 必须正确配置前端静态资源目录。SPA 场景下，`location /` 应使用 `try_files` 做前端路由回退。
4. 实现完成后，至少跑 `pnpm test`。
5. 涉及前端交付时，补跑 `pnpm build:web`。
6. 涉及迁移时，显式跑 `pnpm db:migrate`。
