# 按设计稿实施前端代码

这个技能承接旧的 `fe-figma-gen.run`，负责把设计稿真正变成项目里的页面或组件代码，但不再负责“单独扫描约束”或“单独开向导”。

## 前置条件

- `.specify/memory/fe-figma-gen.md` 已存在且可信
- `.specify/memory/fe-rule.md` 已存在，或当前项目不需要独立前端宪法
- Figma 文件链接或节点链接已经明确
- 如果任务仍处于方案阶段，先使用 `superpowers-writing-plans`

## 工作流

1. 读取 `.specify/memory/fe-figma-gen.md` 与 `.specify/memory/fe-rule.md`。
   先确认目录、命名、样式体系、共享资源和禁止事项。

2. 解析用户给出的 Figma 范围和目标实现范围。
   明确是新页面、局部组件、视觉升级，还是在旧逻辑上做界面还原。

3. 先读取 Figma 设计上下文。
   根据项目可用工具，优先使用 Figma 官方设计上下文，提取结构、变量、截图和复用提示。

4. 必要时准备素材。
   如果需要批量图片、矢量、填充资源或真实文件格式校验，使用 `frontend-figma-toolkit-cli` 处理，不要在主技能里重复定义下载流程。

5. 在写代码前先做组件规划。
   明确哪些部分应该复用现有共享组件，哪些部分应该新建模块内组件，哪些部分需要沉淀为新的共享能力。

6. 按最小必要改动实施。
   页面已存在时，优先保留现有逻辑、状态流和交互；不要因为追求视觉一致而直接重写业务逻辑。

7. 对样式和 token 做真实校验。
   引用的 CSS 变量、颜色 token、尺寸 token 必须在仓库中真实存在；不存在时先回退到本地常量或补充合理定义，不要引用一个空名字。

8. 运行最相关的验证命令。
   至少覆盖类型检查、lint，以及与目标应用相关的测试或构建命令。

## 关键原则

- Figma 是视觉目标，不默认是完整功能说明
- 共享组件优先于重新发明
- 旧逻辑优先于从零重写
- 素材和 token 要真实可落地，不要停留在设计稿语义层
