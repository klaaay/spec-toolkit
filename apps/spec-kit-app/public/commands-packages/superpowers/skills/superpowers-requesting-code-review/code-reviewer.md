## 审查清单

**代码质量：**
- 关注点分离是否清晰？
- 错误处理是否正确？
- 类型安全（如适用）？
- 是否遵循 DRY 原则？
- 是否处理了边缘情况？

**架构：**
- 设计决策是否合理？
- 是否考虑了可扩展性？
- 性能影响如何？
- 是否存在安全问题？

**测试：**
- 测试是否实际测试逻辑（而非模拟）？
- 是否覆盖了边缘情况？
- 需要时是否进行了集成测试？
- 所有测试是否通过？

**要求：**
- 是否满足所有计划要求？
- 实施是否与规范匹配？
- 是否存在范围蔓延？
- 是否记录了重大更改？

**生产就绪性：**
- 迁移策略（如涉及架构更改）？
- 是否考虑了向后兼容性？
- 文档是否完整？
- 是否存在明显错误？

## 关键规则

**应该做：**
- 按实际严重性分类（并非所有问题都是关键）
- 具体明确（文件：行，而非模糊描述）
- 解释问题为什么重要
- 承认优势
- 给出明确裁决

**不应该做：**
- 未检查就说"看起来不错"
- 将吹毛求疵标记为关键
- 对您未审查的代码提供反馈
- 模糊表述（如"改进错误处理"）
- 避免给出明确裁决
