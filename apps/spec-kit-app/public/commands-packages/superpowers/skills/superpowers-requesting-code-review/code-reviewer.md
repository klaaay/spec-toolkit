## 审查清单

**代码质量：**
- 关注点分离是否清晰？
- 错误处理是否正确？
- 类型安全（如适用）？
- 是否遵循 DRY？
- 是否处理了边界情况？

**架构：**
- 设计决策是否合理？
- 是否考虑了可扩展性？
- 是否存在性能影响？
- 是否有安全问题？

**测试：**
- 测试是否验证真实逻辑，而非只验证 mock？
- 是否覆盖了关键边界情况？
- 需要时是否包含集成测试？
- 所有测试是否通过？

**需求一致性：**
- 是否满足所有计划要求？
- 实现是否与规范一致？
- 是否发生了范围蔓延？
- 是否记录了重要变更？

**可交付性：**
- 若涉及迁移，迁移策略是否可行？
- 是否考虑了向后兼容？
- 文档是否足够？
- 是否还存在明显错误？

## 关键规则

**应该做：**
- 按真实严重性分类，不要把所有问题都拔高成关键
- 给出具体文件和行号，不要含糊其辞
- 解释每个问题为什么重要
- 先承认优点，再给裁决
- 给出明确结论

**不应该做：**
- 未检查就说“看起来不错”
- 把吹毛求疵的问题标成关键
- 评论你没有实际审查过的代码
- 用“改进错误处理”这种模糊表述
- 回避明确裁决
