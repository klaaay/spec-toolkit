# 基于条件的等待

## 概览

测试里靠 `setTimeout`、`sleep` 猜时间，通常会制造 flaky test。  
真正稳妥的做法，是等待“你真正关心的条件”出现，而不是猜多久之后它大概会出现。

**核心原则：** 等条件，不等猜测。

## 何时使用

- 测试里出现任意硬编码延时
- 测试偶发失败
- 并行运行时经常超时
- 需要等待异步状态完成

## 核心模式

```typescript
// 错误：猜时间
await new Promise(r => setTimeout(r, 50));

// 正确：等条件
await waitFor(() => getResult() !== undefined);
```

## 常见场景

- 等某个事件出现
- 等某个状态切换
- 等数组长度达到某个值
- 等文件生成
- 等复杂条件同时成立

## 规则

- 轮询间隔不要太短，通常 10ms 就够
- 必须带 timeout
- 循环里每次都取“最新状态”，不要缓存旧值

## 什么时候固定超时是合理的

只有当你在验证**真实定时行为**时，例如 debounce / throttle / 固定 tick 周期，才允许用固定超时；并且要写明：

1. 先等待触发条件成立
2. 固定超时是基于已知时序，而不是拍脑袋
3. 注释里解释为什么需要这个值
