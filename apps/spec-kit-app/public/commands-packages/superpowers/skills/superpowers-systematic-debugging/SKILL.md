# 系统化调试

## 概览

随机修补只会浪费时间，还会引入新 bug。  
“先改改看”通常只是在掩盖症状，而不是解决问题。

**核心原则：** 在尝试任何修复之前，必须先找到根因。只修症状就是失败。

## 铁律

```
未完成根因调查之前，不准提出修复方案
```

如果你还没完成第 1 阶段，就没有资格进入修复。

## 何时使用

适用于任何技术问题：
- 测试失败
- 线上 bug
- 异常行为
- 性能问题
- 构建失败
- 集成问题

**尤其要用在这些场景：**
- 时间压力很大
- “一个快速修复”看起来很诱人
- 你已经试过多个修复
- 上一次修复没生效
- 你其实还没彻底理解问题

## 四个阶段

必须按顺序完成，不能跳。

### 阶段 1：根因调查

在任何修复前：

1. **认真读错误信息**
   - 不要跳过 warning 和 stack trace
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 这个问题能稳定触发吗？
   - 复现步骤是什么？
   - 如果无法稳定复现，就先继续收集信息

3. **检查最近变更**
   - 最近改了什么？
   - 看 git diff、最近提交、依赖或配置变化

4. **对多组件系统加诊断埋点**
   - 在每个组件边界记录输入、输出、环境传播和关键状态
   - 先跑一次收集证据，再判断是哪个层级断了

5. **追踪数据流**
   - 错值最早出现在哪里？
   - 是谁把它传下来的？
   - 一直向上追，直到找到真正起点

相关参考：
- `root-cause-tracing.md`
- `defense-in-depth.md`
- `condition-based-waiting.md`

### 阶段 2：模式分析

在修之前先找模式：

1. 找同代码库里**能工作的类似实现**
2. 若在实现某种模式，先把参考实现**完整读完**
3. 把 working / broken 的差异逐项列出来
4. 理解依赖、配置和环境前提

### 阶段 3：假设与实验

按科学方法做：

1. **一次只提出一个假设**
   - 明确写出来：我认为根因是 X，因为 Y

2. **最小实验验证**
   - 一次只改一个变量
   - 不要同时试多个修复

3. **先验证，再继续**
   - 生效了，才进阶段 4
   - 没生效，就回到阶段 1 形成新假设

4. **如果你其实没看懂**
   - 直接说“我还不理解 X”
   - 不要装懂
   - 去补调查、补阅读、补提问

### 阶段 4：实现

修根因，不修症状：

1. **先做失败用例**
   - 能自动化就自动化
   - 没框架也至少写一次性复现脚本
   - 需要正式失败测试时，用 `superpowers-test-driven-development`

2. **一次只做一个修复**
   - 不要顺手改其他东西
   - 不要绑带式重构

3. **验证修复**
   - 问题是否真正消失？
   - 有没有新回归？

4. **如果修复不生效**
   - 立刻停下
   - 如果失败次数 < 3：回阶段 1
   - 如果失败次数 ≥ 3：开始质疑架构

5. **若 3 次以上修复都失败，开始质疑架构**
   - 每次修复都在不同地方冒出新问题
   - 修复要求“大面积重构”
   - 每次修复都会制造新的症状

这时别继续补丁，先和 human 伙伴讨论是不是整体模式就错了。

## 红旗

如果你脑子里出现这些句子，说明你该停下：

- “先快速修一下，回头再查”
- “先试着改 X 看看”
- “多改几处一起跑测试”
- “先不写测试，手动看看”
- “大概率就是 X”
- “我虽然没完全懂，但这个应该能行”
- “参考实现大概是这个意思，我自己适配一下”
- “主要问题是这些”但你还没完成调查
- 已经失败 2 次，还在说“再试最后一次”

**一旦出现这些念头：回到阶段 1。**

## quick reference

| 阶段 | 核心动作 | 成功标准 |
| --- | --- | --- |
| 1. 根因调查 | 读错误、复现、看变更、收集证据 | 理解发生了什么、为什么会发生 |
| 2. 模式分析 | 找 working case、比差异 | 明确 working / broken 的关键差别 |
| 3. 假设实验 | 单一假设、最小实验 | 假设被证实或被推翻 |
| 4. 实现 | 失败用例、单一修复、验证 | 问题解决，且没有引入回归 |
