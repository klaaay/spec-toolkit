# 根因追踪

## 概览

很多 bug 真正出问题的地方，往往不在报错现场，而是在更早的调用链上。  
如果你只修报错点，那通常只是修了症状。

**核心原则：** 沿调用链向后追，直到找到最初触发点，再在源头修复。

## 适用场景

- 错误发生在调用栈很深的位置
- stack trace 很长
- 不清楚坏数据最初从哪来
- 需要定位到底是哪段测试 / 代码触发了问题

## 追踪步骤

1. **先记录症状**
2. **找到直接触发报错的那行代码**
3. **继续问：是谁调用了它？**
4. **继续往上追传入值**
5. **直到找到最初的错误来源**

## 调试埋点建议

当人工追踪困难时，在危险操作之前加埋点：

- 当前参数
- `process.cwd()`
- 关键环境变量
- `new Error().stack`

在测试里优先用 `console.error()`，比 logger 更稳定。

## 关键原则

**永远不要只修错误出现的地方。**  
先往后追，找到原始触发点，再从源头修。
