# 完成前验证

## 概览

没有新鲜验证证据就宣称“已完成”“已修复”“测试通过”，本质上不是高效，而是不诚实。

**核心原则：** 证据先于结论。

## 铁律

```
没有新鲜验证证据，就不能宣称完成
```

如果你没有在当前轮里运行验证命令，就不能说它“通过了”。

## 闸门函数

在做任何完成性表述之前，必须完成：

1. **识别**：哪个命令能证明这条结论？
2. **执行**：运行完整命令
3. **阅读**：看完整输出、退出码、失败数
4. **确认**：输出是否真的支持你的说法？
5. **然后**：才能下结论

跳过任一步，都是没有验证。

## 常见错误

| 结论 | 需要的证据 | 不足以证明 |
| --- | --- | --- |
| 测试通过 | 测试命令输出为 0 失败 | “应该没问题” |
| Lint 通过 | Lint 输出为 0 错误 | 只看过局部文件 |
| Build 成功 | Build 命令退出码为 0 | 只是 lint 通过 |
| Bug 修复 | 原始症状被验证消失 | 改了代码但没验证 |
| 代理已完成 | 你亲自核验其产物 | 子代理说“成功了” |

## 红旗

如果你脑子里冒出这些话，立刻停下：

- “应该可以了”
- “大概率没问题”
- “先这么说，等会再验”
- “子代理都说好了”
- “我已经很有把握了”

## 适用时机

**以下场景一律适用：**
- 任何“完成 / 修复 / 通过 / 可以合并”表述前
- 任何表示满意或收尾的表述前
- 提交、推送、开 PR 前
- 从一个任务切到下一个任务前

## 底线

先跑命令。  
再看输出。  
最后才下结论。
