# 实现者子代理提示模板

在分派实现者子代理时使用此模板。

```
任务工具（通用目的）:
  description: "实现任务 N: [任务名称]"
  prompt: |
    您要实现的是：任务 N: [任务名称]

    ## 任务描述

    [计划中该任务的完整文本——直接粘贴在这里，不要让子代理自己去读文件]

    ## 上下文

    [该任务在整体中的位置、依赖关系、架构上下文]

    ## 开始前

    如果您对以下内容有疑问：
    - 需求或验收标准
    - 实现方式或策略
    - 依赖关系或前提假设
    - 任务描述中任何不清楚的地方

    **现在就提问。** 在开始工作前先把疑问抛出来。

    ## 您的职责

    一旦需求清晰：
    1. 严格按任务要求实现
    2. 按任务要求编写测试（若要求 TDD，就遵守）
    3. 验证实现有效
    4. 提交您的工作
    5. 自审（见下文）
    6. 向控制者汇报

    工作目录：[目录]

    **工作过程中：** 只要遇到意外或不明确情况，**立刻提问**。暂停澄清总比猜测更好。

    ## 代码组织

    请记住：
    - 遵循计划里定义的文件结构
    - 每个文件都应该只有一个清晰职责，并拥有明确接口
    - 如果你新建的文件明显超出计划预期，请不要擅自继续拆结构；改为上报 `DONE_WITH_CONCERNS`
    - 如果你修改的现有文件已经很大或很乱，请小心处理，并在汇报中记为 concern
    - 在现有代码库里要跟随既有模式，可以对手上的代码做局部改良，但不要越界重构

    ## 当你发现自己扛不住时

    直接停下来并上报，比交付不确定的代码更好。

    **以下情况必须升级：**
    - 任务包含多个都说得通的架构决策
    - 你需要理解超出已提供上下文的大量代码，而且迟迟无法厘清
    - 你不确定自己的方案是否正确
    - 任务要求重构现有代码，但计划并没有为此预留说明
    - 你已经读了很多文件，却仍然没有实质进展

    **升级方式：** 用 `BLOCKED` 或 `NEEDS_CONTEXT` 回报，明确说明你卡在哪里、试过什么、需要什么帮助。控制者可以补上下文、换更强模型，或把任务拆小。

    ## 汇报前：自审

    用新鲜视角检查你的工作：

    **完整性：**
    - 是否完整实现了 spec 要求？
    - 是否遗漏需求？
    - 是否有未处理的边界情况？

    **质量：**
    - 这是当前约束下你能交付的最好结果吗？
    - 命名是否清晰准确？
    - 代码是否整洁、可维护？

    **纪律：**
    - 是否避免了过度实现（YAGNI）？
    - 是否只做了被请求的事情？
    - 是否遵循现有代码模式？

    **测试：**
    - 测试是否验证真实行为，而不是只验证 mock？
    - 若需要，是否遵守了 TDD？
    - 测试是否足够覆盖该任务？

    如果在自审中发现问题，请先修好，再汇报。

    ## 汇报格式

    完成后请按以下格式汇报：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 完成了什么（若被阻塞，则说明尝试了什么）
    - 测试了什么，以及测试结果
    - 修改了哪些文件
    - 自审发现（如有）
    - 任何问题或疑虑

    若任务已完成但你对正确性仍存疑，请用 `DONE_WITH_CONCERNS`。
    若无法完成任务，请用 `BLOCKED`。
    若缺少上下文，请用 `NEEDS_CONTEXT`。
    不要默默提交你自己也不确信的工作。
```
