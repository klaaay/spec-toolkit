## 它做什么

`triage` 会让 issue tracker 中的 issue 穿过一个小型 **state machine**：分类、验证 claim、必要时 grill 成形，并留下 ready-for-agent brief。

它不会盲目打 label。每个被 triage 的条目都必须恰好带有一个 **category** role（`bug` / `enhancement`）和一个 **state** role（`needs-triage`、`needs-info`、`ready-for-agent`、`ready-for-human`、`wontfix`）。它会 _recommend and wait_：告诉你它的 category 和 state 判断及理由，然后只按你的指示行动。在任何东西升为 `ready-for-agent` 前，它会先验证 claim：bug 要复现，PR 要 checkout 并运行。

## 什么时候用

你通过输入 `/triage` 并用自然语言描述目标来调用它。Agent 不会主动调用。例如：“Show me anything that needs my attention”、“let's look at #42”、“move #42 to ready-for-agent”。

当 issue tracker 中有未评估的原始 report，而你希望它们被排序、验证，并转成 Agent 或 human 可领取的 work 时，用它。若要把已稳定的 conversation 转成新 spec，用 [to-prd](https://aihero.dev/skills-to-prd)；若要把已有 PRD 拆成 ticket，用 [to-issues](https://aihero.dev/skills-to-issues)。`triage` 是反方向：处理已经落到 tracker 里的东西。

## 前置条件

`triage` 会读写 issue tracker，所以 [setup-matt-pocock-skills](https://aihero.dev/skills-setup-matt-pocock-skills) 必须先配置 tracker 和 label mapping。上面这些 role name 是 **canonical**；tracker 中真实 label 字符串可能不同，这个映射由 setup 提供。config 还会决定外部 PR 是否算作 request surface，以及谁算 external。

## PR 是带代码的 issue

如果 tracker 把外部 pull request 当作 request surface，`triage` 会让它们穿过 _同一台_ state machine：同样的 category role、同样的 state、同样的 transition。区别只是 state 针对 diff 解读，而不是 report。`ready-for-agent` 表示 brief 已附上，Agent 应该对代码进行下一步；`ready-for-human` 表示可以 merge。Discovery surface 只包含 external PR，但只要显式指定 PR，无论作者是谁都会被 triage。

## Verify before you brief

把 `triage` 和临时打 label 区分开的步骤，是 verification。它会根据 reporter 的步骤复现 bug，或 checkout PR 并运行 tests，然后汇报：confirmed with a code path、failed，或 insufficient detail（这本身就是强 `needs-info` 信号）。它还会运行两项 codebase check：**redundancy**（这是不是已经实现了？如果是，就是 `wontfix`）和 **prior rejection**（`.out-of-scope/` 是否已经拒绝过？）。经过确认的 verification 会形成强得多的 agent brief；猜测不会。

## 它工作正常的信号

- 它触碰的每个条目最后都有恰好一个 category role 和一个 state role；不是零个，也不是两个冲突 state。
- 它给出带理由的 recommendation 并等待，而不是自己直接 relabel。
- bug 被复现，PR 被运行，然后才有东西进入 `ready-for-agent`。
- 它发到 tracker 的每条 comment 都以 `> *This was generated by AI during triage.*` disclaimer 开头。

## 它放在哪里

`triage` 是 issue tracker 的 **periodic maintenance** pass：report 堆起来时运行它，让队列保持有序，让 `ready-for-agent` 栏可信。它位于 tracker 前端，也在 build chain 上游：它写下的 brief 是 [tdd](https://aihero.dev/skills-tdd) 之后会接手实现的内容。当 request 需要打磨时，它依赖 [grilling](https://aihero.dev/skills-grilling) 和 [domain-modeling](https://aihero.dev/skills-domain-modeling) 一次一个问题地把它 grill 成形。近邻是 [to-prd](https://aihero.dev/skills-to-prd)：后者从新 conversation 填充 tracker，而 `triage` 处理已经在 tracker 里的内容。拿不准哪个 Skill 或 flow 适合当前任务时，用 [ask-matt](https://aihero.dev/skills-ask-matt) 路由。
