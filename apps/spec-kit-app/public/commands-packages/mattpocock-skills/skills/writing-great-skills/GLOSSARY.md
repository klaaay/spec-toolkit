# 术语表：构建高质量 Skill

这是 “什么让 Skill 变好” 的领域模型。Skill 的存在，是为了从随机系统中拧出确定性；根本美德是 **可预测性**，下面每个术语都是影响它的杠杆。这是 [`writing-great-skills`](SKILL.md) 披露出的参考资料。

术语按轴线分组：**调用**（Skill 如何被触达）、**信息层级**（内容如何组织）、**引导**（Agent 运行时行为如何成形）和 **删减**（如何保持轻巧）。每个 **失败模式** 都放在能治疗它的杠杆旁边，并标注为 _失败模式_。

任何定义中的 **加粗术语** 也都在本术语表中定义；按标题查找即可。

## 可预测性

Skill 让 Agent 每次运行都以同一种 _方式_ 行动的程度。它要求同样的过程，而不是同样的输出。一个 brainstorming Skill 应该 _可预测地_ 发散：token 会变化，但行为不会。它是所有其他术语服务的根本美德；成本和可维护性是它的症状，不是与它竞争的目标。

_避免使用_：一致性、可靠性、健壮性、输出确定性

## 调用

Skill 如何被触达，以及你会为这个选择支付哪两种负载。

### 模型调用型

保留 **描述** 字段的 Skill。Agent 能看到它并自主触发它，用户也仍然可以手动输入它的名字，所以模型调用型始终 _包含_ 用户可达性。不存在 “仅模型可调用” 的状态：描述只会增加 Agent 发现能力，不会移除人的调用能力。它用每轮永久的 **上下文负载** 换取可发现性。它也能被其他 Skill 触达，因为让它被 Agent 发现的描述，也让它可被调用。内容全是 **参考资料** 的模型调用型 Skill，也可以作为共享参考资料的一个住所：另一个 Skill 能调用它，所以多个 Skill 都需要的参考资料可以放在一处。只有当 Agent 必须自己触达某个 Skill 时才选择模型调用；如果它只会被手动触发，就去掉描述，不支付上下文负载。

_避免使用_：能力、工具、功能

### 用户调用型

移除 **描述** 的 Skill。它对 Agent 不可见，只能由用户输入名字触发。它是用户 _专属_，而 **模型调用型** 是用户 _和 Agent_ 都可达。它用 Agent 不可发现性换取零 **上下文负载**。因为它没有描述，除了用户之外没有任何东西能触达它；其他 Skill 也不能触发它。

_避免使用_：过程、工作流、命令

### 描述

Skill 的机器可读触发器，也是 **模型调用型** Skill 被迫始终加载的一个 **上下文指针**。它的存在本身就是调用轴线：保留它，Skill 就是模型调用型，并能被其他 Skill 触达；删除它，Skill 就是 **用户调用型**，只能由人类触达。它是模型调用型 Skill 的 **上下文负载** 来源。

_避免使用_：frontmatter、摘要

### 上下文指针

上下文中持有的引用：它命名某个上下文之外的材料，并编码触达它的条件。**描述** 是顶层上下文指针（上下文窗口 -> Skill）；指向披露文件的指针也是同一个对象的下一层。决定 Agent _何时_ 触达以及 _多可靠_ 触达的是指针措辞，不是目标。必须触达的材料如果藏在措辞薄弱的指针后面，就是方差 bug：先修指针措辞；只有修措辞无效时，才把材料内联回来。

_避免使用_：链接、引用、导入

### 上下文负载

**模型调用型** Skill 对 Agent 上下文窗口施加的成本：它的 **描述** 永远加载，占用 token 和注意力。**用户调用型** Skill 通过没有描述来避开它。它也是拆出更多模型调用型 Skill 时的刹车。

_避免使用_：token 成本、上下文膨胀

### 认知负载

**用户调用型** Skill 对人类施加的成本：用户必须记住有哪些 Skill，以及什么时候该触发哪个。人类就是索引。**模型调用** 通过让 Agent 可发现来移除这项成本。它也是拆出更多用户调用型 Skill 时的刹车。认知负载不是必须最小化的东西，它是人类能动性的价格，也是某些 Skill 应保持用户调用型的原因。把它花在人类判断重要的地方；在不重要的地方移除它。

_避免使用_：人类索引、负担、开销

### 路由 Skill

一种 **用户调用型** Skill，职责是指向其他用户调用型 Skill：列出每个 Skill，并说明什么时候该触发哪个。这样用户只需要记住一个 Skill，而不是很多个。它只能提示，不能真正触发它们：用户调用型 Skill 没有 **描述**，所以除了人类外没有任何东西能触达它们。当用户调用型 Skill 变多时，路由 Skill 用来治理 **认知负载**。

_避免使用_：分发器、菜单、注册表、索引、路由过程

### 粒度

你把 Skill 切分得多细。更细的切分会消耗两种负载之一：更多 **模型调用型** Skill 消耗 **上下文负载**，也就是更多描述挤在窗口中并争夺注意力；更多 **用户调用型** Skill 消耗 **认知负载**，也就是用户要记住并触达更多东西。两种切法指导粒度。按 **调用** 切分：当你有一个不同的 **引导词** 能独立触发，而且这个触发词确实出现在 prompt 中，就拆出模型调用型 Skill。按 **序列** 切分：当某个步骤的 **完成后的步骤** 需要被隐藏时，把一串 **步骤** 拆开，因为隔离到自己的上下文可以清掉后续内容。也要警惕反向问题：合并序列会把每个步骤的后续内容暴露给前面的步骤，诱发过早完成。

_避免使用_：切块、模块化

## 信息层级

Skill 内容如何组织，以及每一块内容放在梯子的多深处。

### 信息层级

Skill 内容按 Agent 多快需要它来排序后形成的单一梯子，由两种切法产生：在文件内还是指针后，步骤还是参考资料。梯级如下：

- **步骤**：文件内，主层级。
- **参考资料**：文件内，次层级。
- **参考资料**：已披露，位于 **上下文指针** 后面。

没有 **步骤** 的 Skill 只使用后两级，常见情况是一个合理的扁平同级集合，例如评审规则全在一层。这种安排没问题，不是坏味道。信息层级和调用方式无关：一个 Skill 可以是模型调用型或用户调用型，也可以全是步骤、全是参考资料，或两者都有。当 Skill 有步骤时，本该披露出去的文件内参考资料会埋掉步骤，让 Agent 是否注意到它们变成抛硬币。这是方差杠杆，不只是可读性问题。保持梯子顶部清晰；能下推的内容就下推。

_避免使用_：结构、组织、布局

### 步骤

Agent 执行的有序动作。只要 Skill 有步骤，它们就是内容的主层级，也是值得放进 `SKILL.md` 的部分。并非每个 Skill 都有步骤：Skill 可以全是步骤（如 `tdd`）、全是 **参考资料**（如评审），或两者都有，且不受调用方式影响。每个步骤都以一个清晰或模糊的 **完成标准** 结束。

_避免使用_：工作流、指令、编排

### 参考资料

Agent 按需查阅的材料：定义、事实、参数、示例、条件性指令。Skill 有 **步骤** 时，它低于步骤；Skill 没有步骤时，它就是全部内容；它也可以完全位于 Skill 之外，见 **外部参考资料**。参考资料通过 **上下文指针** 触达，是 **渐进披露** 的主要候选。

_避免使用_：支持材料、文档、背景

### 外部参考资料

位于 Skill 系统之外的 **参考资料**：普通文件，没有 **描述**，没有 **步骤**，不可被调用，但任何 Skill 都可以指向它。它适合承载不需要自己触发的共享参考资料，也是两个 **用户调用型** Skill 共享资料的唯一共同住所，因为它们都没有描述，不能互相触发。

_避免使用_：文档、资源、知识库

### 渐进披露

把 **参考资料** 沿梯子下移：从 `SKILL.md` 移到 **上下文指针** 后面，让顶层保持清晰。它首先不是 token 优化，而是保护 **信息层级** 的方法。它由 **分支** 授权：只被部分分支需要的内容披露出去，每条路径都需要的内容内联。如果必须触达的材料后面的指针不可靠，先强化指针措辞；只有强化失败时，才把材料拉回内联。

_避免使用_：懒加载、切块

### 就近放置

把 Agent 需要同时理解的材料放在一处：一个概念的定义、规则和注意事项放在同一个标题下，而不是散落在文件各处。这样读到一部分时，相关邻居也会一起进入上下文。它是 **信息层级** 在文件内的搭档：信息层级决定一块内容 _放得多深_，就近放置决定它到了那里之后 _和什么放在一起_。参考资料正文没有固定格式；测试标准是 Skill 是否像写给 Agent 的文档。分组材料像文档，散落材料不像。它不同于 **重复**：重复是一个含义在两个位置出现；散落是一个含义被拆成碎片，分布在多个位置。

_避免使用_：分组、聚类、内聚

### 蔓延

_失败模式。_ Skill 单纯过长，也就是 `SKILL.md` 中行数太多。它不取决于内容是否陈旧或重复。即使每一行都还活着、都唯一，也可能蔓延。它消耗可读性：Agent 行动前要淌过更多内容，注意力被多余内容摊薄；也消耗可维护性：每多一行，就多一个需要保持 **相关** 的位置；还消耗 token。解法是 **信息层级**：把 **参考资料** 推到 **上下文指针** 后面，并按 **分支** 或序列拆分，让每条路径只携带自己需要的东西。它不同于 **沉积**（陈旧累积造成的长度）和 **重复**（重复含义造成的长度）。蔓延就是长度本身，不管成因是什么。

_避免使用_：膨胀、长度、体量、冗长

## 引导

这些杠杆让 Agent 运行时行为朝 **可预测性** 成形。

### 分支

Skill 可以被调用的一种不同方式，也就是 Skill 处理的一类情况。不同运行会沿着不同路径走。一个步骤很多的 Skill 可能有很多分支；线性 Skill 则没有。

_避免使用_：路径、情况、分叉

### 引导词

一个紧凑概念，也可以称为 _Leitwort_。它已经存在于模型预训练中，Agent 运行 Skill 时会借它思考。它用最少 token 编码一个行为原则，因为它调用了模型已有先验，例如 _lesson_、_proximal zone of development_、_fog of war_、_tracer bullets_。它以 token 形式重复，而不是以句子形式重复，并在 Skill 中逐渐积累分布式定义，锚定一整片行为。你可以创造自己的词，只要定义清楚；但自造词没有先验可招募，你要用定义 token 支付本可由预训练词免费提供的东西。优先寻找已有词。

引导词两次服务 **可预测性**。在正文中，它锚定 **执行**：每次这个概念出现，Agent 都会触达相同行为；在扁平参考资料中，它会把注意力聚焦到一类要检查的东西上，每次运行都招募相同检查。在 **描述** 中，它锚定 **调用**，而且不仅限于 Skill 内部：当同一个词存在于你的 prompt、文档和代码库中时，Agent 会把这种共享语言和 Skill 关联起来，更可靠地触发它。用你实际希望触发 Skill 时会使用的引导词来写描述。

_避免使用_：关键词、术语、母题

### 完成标准

告诉 Agent 一个工作单元已经完成的条件，也就是它用来判断的目标。两个属性让它成为杠杆，而不只是质量要求。第一是 **清晰度**：Agent 能否判断完成和未完成？它抵抗 **过早完成**。模糊边界，例如 “达成理解”，会让 Agent 宣称完成并滑向下一步。这个轴线需要 _步骤_ 才能生效，因为过早完成是步骤之间的失败。第二是 **要求程度**：它要求多少工作？它设定 **摸排工作**。例如 “每个被修改的模型都已说明” 会强制充分工作，而 “产出变更列表” 不会。这个轴线 _不依赖步骤_：它也能约束扁平参考资料，这就是没有步骤的 Skill 仍能设置穷尽性门槛的方式，例如 “每条规则都已应用”。最强的完成标准既可检查，也足够穷尽。

_避免使用_：完成条件、退出条件、停止规则

### 摸排工作

Agent 在单个步骤内部幕后完成的工作：读文件、探索代码库、修改内容、挖掘所需信息，而不是把问题推给用户。它位于步骤结构之下，从不写成独立步骤，而是潜伏在措辞中，由 Agent 控制，不由 Skill 控制。它是 **完成后的步骤** 跨步骤拉力在步骤内部的对应物。它可以由 **引导词** 提高，例如 _comprehensive_、_thorough_，也可以由要求穷尽的 **完成标准** 提高，包括把要求轴线应用到扁平参考资料上，这会驱动参考型 Skill 覆盖所有层级。当要求缺失，或 **过早完成** 截断步骤时，摸排工作会变薄。

_避免使用_：范围、努力、勤勉、覆盖

### 完成后的步骤

当前步骤后面的 **步骤**。如果它们可见，就会把 Agent 往前拉，导致 **过早完成**。Agent 看到得越多，拉力越强。防御方式是通过拆分序列隐藏它们。

_避免使用_：视野、fog of war、前瞻

### 过早完成

_失败模式。_ 当前步骤还没有真正完成，Agent 的注意力就滑向 “已经完成”，于是提前结束。这是步骤之间的失败：它需要 **步骤** 才会发生。没有步骤的 Skill 过早退出，不叫过早完成，而是要求未满足导致的 **摸排工作** 变薄。它是两股力量的拉扯：可见的 **完成后的步骤** 往前拉，**完成标准** 的清晰度提供阻力。清晰、可检查的门槛能守住；模糊门槛会让位。模糊是必要条件：清晰边界可以抵抗拉力，不管后面有多少步骤可见，所以从不抢跑的步骤不需要防御。确实会抢跑的步骤有两个杠杆，但要按顺序使用：**先收紧边界**，因为这局部且便宜。只有当标准不可避免地模糊，而且你确实观察到抢跑时，才 **隐藏后续步骤**。隐藏只有跨真实上下文边界才有效，例如用户调用型 handoff 或子 Agent 分派；内联的模型调用型调用仍会让后续步骤留在上下文里，无法清掉它们。过早完成会造成摸排工作变薄，但二者不同：即使步骤执行到完整结束，摸排工作也可能仍然很薄。

_避免使用_：premature closure、抢跑、匆忙、走捷径

## 删减

让 Skill 保持轻巧。每个疗法都对应一个失败模式。

### 单一事实源

每个含义恰好有一个权威位置。这样修改 Skill 行为时，只需要改一个地方。**重复** 是它的反面。

_避免使用_：归属、规范位置

### 重复

_失败模式。_ 同一个含义拥有超过一个 **单一事实源**。它消耗维护成本：改一处就必须改其他处；消耗 token；还会抬高显著性，让这个含义在层级上的权重超过实际排名。它是 **引导词** 的意外反面：引导词通过重复 token 有意提高注意力，但不重复含义。

_避免使用_：反复、冗余

### 相关性

一行内容是否仍然影响 Skill 要做的事。这是决定保留什么的镜头。一行会失去相关性，可能是因为它从来不影响任务，例如单纯阐述，或本应披露出去的 **分支**；也可能是它过期了，随着行为或世界变化而漂移。更短的 Skill 更容易保持相关性，因为每一行检查起来更便宜。它不同于 **无效指令**：相关性问的是这一行是否影响任务，而不是它是否改变行为。

_避免使用_：承重、新鲜度、时效

### 沉积

_失败模式。_ 旧内容层层沉在 Skill 中，从不被清理。原因是添加看起来安全，删除看起来有风险，于是陈旧、无关的行不断堆积。你必须往下钻，才能找到仍然有效的部分。这是任何没有删减纪律的 Skill 的默认命运；它是 **相关性** 的缓慢侵蚀，不同于 **重复** 的含义反复。

_避免使用_：累积、膨胀、垃圾、腐化

### 无效指令

_失败模式。_ 一条不会改变行为的指令，因为模型默认就会这么做。你支付负载，却只是告诉 Agent 它本来就会做的事。测试方式：这行相比默认行为有没有改变？一行可以非常 **相关**，但仍然是无效指令。让 **引导词** 免费生效的同一组先验，也让无效指令变得毫无价值。

引导词是一种 _技术_；无效指令是对一行内容的 _判决_，二者会交叉。弱到无法超越默认行为的引导词，就是无效指令，例如当 Agent 本来就还算 thorough 时写 _be thorough_。修复方式是换一个能通过判决的更强词，例如 _relentless_，而不是换另一种技术。因此，无效指令测试，也就是 “相比默认行为有没有改变”，也是评估引导词是否值得重复的方式。这是相对于模型而言，不是相对于读者而言：两个人争论一行是不是无效指令，本质是在争论默认行为是什么。解决方式是运行这个 Skill，而不是辩论。

_避免使用_：冗余指令、重复显然内容、反复强调
