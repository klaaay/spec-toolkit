Skill 的存在，是为了从随机系统中拧出确定性。**可预测性** 是根本美德：Agent 每次运行都采取同样的 _过程_，而不是产出同样的结果。下面所有杠杆都服务于它。

**加粗术语** 都在 [`GLOSSARY.md`](GLOSSARY.md) 中定义；需要完整含义时去那里查。

## 调用方式

有两种选择，它们支付不同成本：

- **模型调用型** Skill 保留 **描述**，让 Agent 能自主触发它，其他 Skill 也能触达它（用户仍然可以手动输入它的名字）。它会增加 **上下文负载**，因为描述每一轮都待在上下文窗口里。机制：不要写 `disable-model-invocation`，并写一段面向模型的描述，包含丰富触发语，例如 “Use when the user wants..., mentions...”。
- **用户调用型** Skill 把描述从 Agent 可达范围里拿掉：只有用户输入它的名字时才能调用，其他 Skill 也不能调用。它没有上下文负载，但会消耗 **认知负载**：_用户_ 成为索引，必须记得它存在。机制：设置 `disable-model-invocation: true`；此时 `description` 只面向人类，是一句去掉触发列表的摘要。

只有当 Agent 必须自己触达某个 Skill，或另一个 Skill 必须触达它时，才选择模型调用。如果它永远只会被手动触发，就做成用户调用型，不支付上下文负载。

当用户调用型 Skill 多到记不住时，堆积的认知负载要用 **路由 Skill** 治理：用一个用户调用型 Skill 列出其他 Skill，以及什么时候该用哪个。

## 编写 description

模型调用型 **描述** 做两件事：说明这个 Skill 是什么，并列出应该触发它的 **分支**。每个词都会增加 **上下文负载**，所以描述比正文更需要狠剪：

- **把 Skill 的引导词放在前面。** 描述正是在这里完成调用工作。
- **一个分支只写一个触发。** 如果多个同义说法只是重命名同一个分支，那就是 **重复**。例如 “build features using TDD ... asks for test-first development” 是同一个分支写了两遍。合并它们，只保留真正不同的分支。
- **删掉正文里已经有的身份说明。** 描述只保留触发条件，以及任何 “当另一个 Skill 需要……” 的触达子句。

## 信息层级

Skill 由两类内容组成：**步骤** 和 **参考资料**。它们可以自由混合：一个 Skill 可以全是步骤，全是参考资料，或两者都有。核心决策是用哪一类，以及把它放在 **信息层级** 的哪一层。信息层级是一架梯子，按 Agent 多快需要这些材料来排序：

1. **Skill 内步骤**：`SKILL.md` 中的有序动作，是主层级，说明 Agent 按什么顺序做什么。每个步骤都以一个 **完成标准** 结束，也就是告诉 Agent 工作已经完成的条件。让它 _可检查_，Agent 能判断完成和未完成；在重要场景里还要 _穷尽_，例如 “每个被修改的模型都已说明”，而不是 “产出一个变更列表”。模糊标准会诱发 **过早完成**。
2. **Skill 内参考资料**：`SKILL.md` 中的定义、规则或事实，按需查阅。很多时候它们可以是合理的扁平同级集合，例如评审规则全放在同一层，这不是坏味道。_本 Skill 就全是参考资料。_
3. **外部参考资料**：从 `SKILL.md` 移到单独文件里的参考资料，通过 **上下文指针** 触达，只在指针触发时加载。它可以是 _已披露_ 的参考资料，例如同目录的 `GLOSSARY.md`，仍属于这个 Skill；也可以是完全的 **外部参考资料**，位于 Skill 系统之外，任何 Skill 都可以指向它。

高要求的完成标准会驱动充分的 **摸排工作**，也就是 Agent 在任务内部做的挖掘。无论 Skill 是否有步骤都是如此，因为 “每条规则都已应用” 对扁平参考资料的约束，和 “每个步骤都已完成” 对序列的约束一样有效。

往下推得太少，顶层会膨胀；往下推得太多，又会藏起 Agent 实际需要的材料。这个张力就是全部决策。

**渐进披露** 是沿着梯子向下移动：把内容从 `SKILL.md` 移到链接文件中，让顶层保持清晰。机制：在 Skill 文件夹中放一个链接的 `.md` 文件，文件名说明它包含什么。本 Skill 把完整定义披露到 `GLOSSARY.md`。有些 Skill 有多种用法，每一种不同用法都是一个 **分支**，也就是不同运行会沿着不同路径走。分支是最干净的披露测试：每个分支都需要的内容内联，只被部分分支触达的内容放到指针后面。**上下文指针** 的 _措辞_，而不是它的目标，决定 Agent 何时以及多可靠地触达材料。

如果说梯子决定一段内容 _放得多深_，**就近放置** 决定内容到了那里后 _和什么放在一起_：把一个概念的定义、规则和注意事项放在同一个标题下，而不是散落各处。这样读到一部分时，相关邻居也会一起进入上下文。

## 什么时候拆分

**粒度** 是你把 Skill 切得多细。每次切分都会消耗两种负载之一，所以只有当切分值得时才切。两种切法：

- **按调用拆分**：当你有一个独立 **引导词** 应该自己触发，或另一个 Skill 必须触达它时，拆出一个 **模型调用型** Skill。你会为新的常驻 **描述** 支付 **上下文负载**，所以独立触达必须足够有价值。
- **按序列拆分**：当一串 **步骤** 后面尚未执行的内容，也就是某个步骤的 **完成后的步骤**，会诱使 Agent 匆忙跳过当前步骤而 **过早完成** 时，拆分这段序列。把后面的步骤从视野中移走，会鼓励 Agent 在当前任务上做更多 **摸排工作**。

## 删减

让每个含义都有一个 **单一事实源**：一个权威位置。这样修改行为时只需要改一处。

逐行检查 **相关性**：这一行是否仍然影响 Skill 要做的事？

然后逐句寻找 **无效指令**，不要只按行检查。把每句话单独拿出来做无效指令测试。一旦失败，删除整句话，而不是只修剪几个词。要果断：大多数失败的 prose 应该被删除，而不是改写。

## 引导词

**引导词** 是一个紧凑概念，已经存在于模型预训练中，Agent 在运行 Skill 时会借它思考，例如 _lesson_、_fog of war_、_tracer bullets_。它在文本中反复出现（强引导词不一定需要反复出现），逐渐积累分布式定义，并用最少 token 锚定一整片行为，因为它调动了模型已有先验。

它从两个方面服务可预测性。在正文中，它锚定 _执行_：每次这个词出现，Agent 都会触达同一类行为。在描述中，它锚定 _调用_：当同一个词存在于你的 prompt、文档和代码中时，Agent 会把这种共享语言和 Skill 关联起来，更可靠地触发它。

主动寻找机会，把 Skill 重构到引导词上。一个三元组在三个地方展开说明（**重复**），一个描述花一句话指向一个想法，这些段落都在请求被 **压缩** 成一个 token。例子：

- "fast, deterministic, low-overhead" -> _tight_：把某个阶段反复出现的一组质量压缩成一个预训练词，例如一个 _tight_ loop。
- "a loop you believe in" -> _red_：把模糊门槛转成二元可观察状态，这个循环要么在 bug 上变 _red_，要么没有。

收益有两层：token 更少，Agent 也有更锋利的钩子挂住思考。假设每个 Skill 都携带着可以被引导词退休的重复表述，然后去找它们。

## 失败模式

用这些模式诊断用户在使用 Skill 时可能遇到的问题。

- **过早完成**：步骤还没真正做完，就因为注意力滑向 “已经完成” 而结束。防御顺序：先收紧完成标准，这便宜且局部；只有当标准不可避免地模糊，并且你观察到 Agent 真的会抢跑时，才通过拆分隐藏完成后的步骤。
- **重复**：同一个含义出现在多个位置。它增加维护成本和 token，并把某个含义在层级上的权重抬得高于实际重要性。
- **沉积**：陈旧层不断沉下来，因为添加看起来安全，删除看起来有风险。任何没有删减纪律的 Skill，默认都会走向这里。
- **蔓延**：Skill 单纯过长，即使每一行都还活着、都不重复，也仍然过长。它伤害可读性和可维护性，并浪费 token。解法是信息层级：把 **参考资料** 披露到指针后面，并按 **分支** 或序列拆分，让每条路径只携带自己需要的东西。
- **无效指令**：模型默认已经会遵守的行，所以你付出了负载却没有改变行为。测试方式：这句话相比默认行为有没有改变结果？弱引导词也是无效指令，例如当 Agent 本来就还算 thorough 时，再写 _be thorough_。修复不是换技术，而是换成更强的词，例如 _relentless_。
