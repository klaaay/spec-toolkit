# MISSION.md 格式

`MISSION.md` 位于工作区根目录。它记录用户学习这个主题的 _原因_。每一个教学决策，包括接下来教什么、推荐哪些资源、设计哪些练习，都应该追溯到这份文档。

## 模板

```md
# 使命：{主题}

## 为什么

{1-3 句话。用户追求的具体真实世界目标。掌握这项技能后，他们的生活或工作会发生什么变化？避免 “理解 X” 这种抽象表述，要追问背后的结果。}

## 成功标准

- {用户将能做到的一件具体、可观察的事}
- {另一件具体的事}
- {……}

## 约束

- {时间、预算、已有承诺、学习偏好，或任何限制教学方式的条件}

## 范围外

- {用户明确暂时不想追的相邻主题，用来保护最近发展区}
```

## 规则

- **一个工作区只放一个使命。** 如果用户想学两个无关主题，那就是两个工作区。
- **具体胜过抽象。** “10 月前跑完半马” 胜过 “变健康”。“给团队交付一个 Rust CLI” 胜过 “学 Rust”。
- **对模糊表达追问。** 如果用户说不清为什么，先采访他们，再写任何内容。糟糕的使命比没有使命更糟。
- **现实变化时就修订。** 使命会改变。当用户目标移动时，更新这个文件，不要让过期使命继续引导后续课程。
- **保持短小。** 如果 `MISSION.md` 超过一屏，它就不再是指南针，而变成了计划。
