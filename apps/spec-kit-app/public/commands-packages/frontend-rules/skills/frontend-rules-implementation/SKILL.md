# 按前端宪法实施

这个技能承接旧的 `fe-rule.run`，但不再把“写计划”和“开始实施”混在一起。现在的边界是：

- `superpowers` 负责把方案写成已确认的实施文档
- `frontend-rules-implementation` 负责读取前端宪法与实施文档，按现有工程约束真正落地代码

## 前置条件

- `.specify/memory/fe-rule.md` 已存在且可信
- 当前任务已有已确认的实施文档，或用户给出的实现目标已经足够具体

如果前端宪法缺失，先使用 `frontend-rules-bootstrap`。
如果实施文档还没写好，先使用 `superpowers-writing-plans`。

## 实施输入优先级

读取上下文时，按下面的优先顺序理解本次任务：

1. 当前用户消息
2. 已确认的实施文档或任务拆解
3. `tasks.md`、`plan.md`、`.specify/memory/plan.md` 等计划产物
4. 仓库内和当前模块直接相关的补充文档

## 工作流

1. 读取 `.specify/memory/fe-rule.md`，提取本次实现真正相关的约束。
   不要把整份规则文档机械复述到回答里，只抽取影响当前改动的部分。

2. 读取已确认的实施文档，确认影响范围。
   包括目标页面、模块、共享组件、状态流、接口依赖、测试要求与校验命令。

3. 在写代码前，先把改动边界与文件落点对齐到前端宪法。
   如果计划与宪法冲突，先解释冲突点，再决定是修正计划还是局部偏离。

4. 按最小必要改动实施。
   优先复用现有目录、抽象和组件；只有现有结构确实无法承载需求时，才新增抽象。

5. 对共享影响做显式控制。
   任何涉及共享组件、公共 hook、状态容器、网络层封装的改动，都要先确认是否会影响其他模块。

6. 完成后运行最相关的验证命令。
   至少覆盖格式化、类型检查、lint、以及本次改动相关的测试或构建。

7. 汇报时说明三件事：
   - 本次改动如何遵循前端宪法
   - 哪些地方因为现实原因做了局部偏离
   - 还剩哪些风险或待确认项

## 关键原则

- 不重新设计需求，只做计划内的前端落地
- 不因为“顺手”扩散到与任务无关的重构
- 不新增和宪法冲突的目录、命名或依赖
- 若现有规则过期，应先刷新前端宪法，再继续实施
