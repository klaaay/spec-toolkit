# 建立前端宪法

通过扫描仓库中的真实证据，建立一份稳定、可执行、可被后续前端技能复用的工程规则文档。这个技能吸收了旧的 `fe-rule.scan` 和 `fe-rule.wizard` 两条路径：先自动扫描，再只针对缺口追问，而不是把扫描和向导拆成两个入口。

## 目标产物

- `.specify/memory/fe-rule.md`

## 职责边界

- 负责：前端工程规则建模
- 不负责：Figma 落地、接口契约实现、项目业务需求设计
- 如果用户还在讨论需求方案，先回到 `superpowers-brainstorming`

## 工作流

1. 先读取现有的 `.specify/memory/fe-rule.md`。
   如果文件已存在，不要盲目重写；先判断是增量刷新还是整体过期。

2. 基于仓库证据做自动扫描。
   优先看顶层配置，再看目标应用或工作区，避免只从单个文件得出草率结论。

3. 重点识别这些维度：
   - 包管理器与 monorepo 结构
   - 前端框架、路由方案、运行时约束
   - UI 组件库、样式体系、Design Token 来源
   - 图标库与资源组织方式
   - HTTP 客户端、服务层模式、错误处理方式
   - 全局状态和服务器状态方案
   - lint、格式化、测试和构建命令
   - 目录结构、命名模式、文件落点习惯

4. 证据不足时，再向用户补最少的问题。
   不要把技能退化成纯问答式 wizard。只有自动扫描无法确认的关键项，才追问用户。

5. 输出简洁、可执行的规则文档。
   优先写“该怎么做”，而不是写长篇背景解释。每条关键结论尽量附带证据来源或理由。

6. 对未知项显式标记。
   统一写成 `未知：[原因]`，不要伪装成确定结论。

7. 将结果写入 `.specify/memory/fe-rule.md`。
   默认覆盖旧内容，但应保留仍然有效的规则表达，不要因为重写而丢掉清晰的既有结论。

## 输出要求

- 文档以规则为中心，避免散文化描述
- 规则必须能直接指导后续代码实现
- 结论优先基于仓库证据，而不是基于技术偏好
- 如果存在多个候选路径，要写清推荐项和理由

## 交付提示

完成后，明确告诉用户：

- 前端宪法已经写入 `.specify/memory/fe-rule.md`
- 如果接下来要开始实现，应先确保实施文档已经确认，然后再使用 `frontend-rules-implementation`
