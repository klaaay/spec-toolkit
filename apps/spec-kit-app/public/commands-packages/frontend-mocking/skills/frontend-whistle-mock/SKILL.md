# 使用 Whistle 建立前端接口 Mock

通过 Whistle 规则和本地 JSON 数据，快速创建可重复使用的接口 Mock。这个技能负责通用流程，不绑定具体业务项目；如果某个项目有自己的命名约定、默认返回结构或目录组织，可以在项目 skill 里基于本技能再包一层。

## 工作流

1. 先读取 `.specify/configs.json`。
   如果文件不存在，就创建目录和文件。至少检查下面两项配置：
   - `whistle-mock-base-dir`
   - `local-dev-url`

2. 缺失配置时再向用户追问。
   只问必要配置，不要把技能变成长向导。

3. 从项目根目录读取 `package.json` 的 `version`。
   这个版本号默认作为 mock 目录的版本标识。

4. 解析用户输入。
   至少识别：
   - 接口功能描述
   - 接口请求路径
   - 期望的 Mock 返回数据

5. 生成规则名和 mock 文件名。
   统一使用 kebab-case 的功能名，避免规则名和文件名难以维护。

6. 创建或更新 Whistle 规则。
   规则内容至少要包含：
   - 匹配路径
   - 对应的本地 mock 文件路径
   - 成功状态码

7. 创建或更新本地 mock 数据。
   数据文件放在 `{whistle-mock-base-dir}/v{version}/` 下，保持版本隔离。

8. 创建或更新对应的 Whistle value。
   让 Whistle 面板内也能直接看到和编辑当前响应内容。

9. 启用规则并核验状态。
   创建后要确认规则已经启用，而不是只写了文件。

10. 汇报最终结果。
    至少返回：

- 规则名称
- 命中的接口路径
- 本地 mock 文件路径
- 当前启用状态

## 通用约束

- 路径命中规则要避免互相覆盖
- JSON 文件统一用 UTF-8 和稳定缩进
- 如果规则已存在，优先更新而不是重复创建近似规则
- 若用户给的返回数据不是合法 JSON，先停下来要求修正

## 适合项目层二次封装的地方

- 默认返回结构，例如是否必须有 `code`、`message`、`data`
- 规则命名约定
- 特定项目的本地开发地址和 mock 根目录约定
- 某些接口路径的 baseUrl 拼接方式
